# WORKFLOW.md — 自動作業フロー

## セッション開始手順

**読み込み順序は `CLAUDE.md` で定義**。本ファイルはその中の1つとして読まれる。

参照資料（必要に応じて）：
- `docs/design/current/clabotch_design_doc_v11.md` — v11最終設計書（常に最新版を参照）

コードとドキュメントが矛盾する場合は、設計書 v11 を優先し、古いドキュメントを更新する。

---

## 作業フロー: Explore → Plan → Code → Commit

### 1. Explore（探索）

- いきなりコードを書かない
- 既存コード・ファイル構造を先に理解する
- 設計書 v11 §14（実装ロードマップ）を起点にする

### 2. Plan（計画）

- 実装前に詳細な計画を立てる
- タスクを小さな単位に分解する
- リスクと依存関係を事前に特定する
- 計画を `docs/exec-plans/active/` に記録する

### 3. Code（実装）

- TDD優先：テストを先に書いてから実装する
- 1機能ずつ実装してテストを通過させる
- 独立したタスクは並列実行可
- 実装しながら随時 `swift build` / `xcodebuild` で確認

### 4. Commit（コミット）

- 1機能1コミットを原則とする
- コミットメッセージは変更内容を具体的に記述する
- `git push --force` は禁止

## 計画フェーズのルール

実装前に計画を立てた場合、リポジトリルートで以下を実行すること：

```bash
# リポジトリルートで実行
codex exec --full-auto "以下の実装計画をシニアエンジニアの視点でレビューしてください。技術的なリスク・抜け漏れ・改善提案を日本語で指摘してください。テストや実装は不要です。計画のテキスト分析のみ行ってください：

[計画内容をここに貼る]"
```

Codexの計画レビューでOKが出てから実装を開始すること。

## 品質ゲート

作業完了前に必ず確認：

1. `swift build` がエラーゼロで終了する
2. `xcodebuild test` が通過する（Unit Tests）
3. ユニットテスト可能なロジックには必ずテストを追加する
4. 実装後はCodexにレビューを依頼し、A以上になるまで修正を繰り返す

## Auto Continue（自動継続ポリシー）

各フェーズ完了後、停止条件がなければ次の最優先タスクへ自動で進む。

### 実行サイクル

1. 現状確認（HANDOVER.md の backlog を読む）
2. 最優先タスクを自動選択
3. 必要なら実装計画作成 → Codex 計画レビュー A 取得
4. 実装 → build/test → Codex 実装レビュー A 取得 → 必要なら修正
5. コミット → push
6. 計画を `completed/` に移動
7. HANDOVER.md 更新（backlog の優先度を再評価）
8. 次の最優先タスクへ継続

### 優先度ルール（上が高い）

1. 既存バグ修正
2. 回帰防止テスト
3. テスト容易化・DI・保守性改善
4. 小規模ポリッシュ
5. 調査のみの項目
6. 外部依存で止まる項目

### 停止条件

以下の場合のみユーザーに確認して停止する：

- 権限承認が必要
- 外部ソフト導入やユーザー操作が必要
- 複数の active job があり選択不能
- 要件矛盾または高リスク判断が必要
- ユーザーが明示的に停止を要求した

### backlog の管理

- HANDOVER.md の「未完了」セクションを正とする
- 各サイクル完了後に backlog を更新する
- 優先度は上記ルールに従い自動で並べ替える
